# token防爆破？ 1.随便输入账密后抓包，发送到intruder模块，账号，密码，token添加payload，使用PItchfork类型 2.在设置里找到检索-提取，依次点击添加，获取响应，（选中token的值）确定 ![image-20251114203321501](C:\Users\23214\AppData\Roaming\Typora\typora-user-images\image-20251114203321501.png) 3.找到重定向，点击总是 ![image-20251114201439674](C:\Users\23214\AppData\Roaming\Typora\typora-user-images\image-20251114201439674.png) 4.找到攻击结果，取消勾选生成未修改的基本请求 ![image-20251114201456169](C:\Users\23214\AppData\Roaming\Typora\typora-user-images\image-20251114201456169.png) 5..在资源池中新建资源池，设置最大请求为1，防止请求过多token还没用就被刷新 ![image-20251114201517456](C:\Users\23214\AppData\Roaming\Typora\typora-user-images\image-20251114201517456.png) 6.重新刷新一次浏览器，找到token 7.token payload选择递归提取，把刚得到的token输进去，爆破 ![image-20251114215416921](C:\Users\23214\AppData\Roaming\Typora\typora-user-images\image-20251114215416921.png) # 注： 1.token： 每次请求登录页面时，服务器会生成一个唯一的 Token，并将其嵌入到页面中的隐藏字段。提交登录请求时，必须携带该 Token，否则服务器拒绝处理请求。 后端先校验 token： 不存在 → 直接 403/302，不校验账号密码。 存在 → 才继续走账号密码逻辑。 2.bp中Intruder 的默认行为 跑字典时，Intruder 只会机械地替换 §user§、§pass§。导致第二次即之后的爆破并不携带token ，因此全部 403。 3.检索-提取，把 HTML 里 `value="..."` 那段剪出来，临时存到内存变量 §token§。 4.总是重定向，让 Burp 把 302 跟到底，确保剪 token 时拿到的是最终 HTML 5.取消生成未修改的基本请求,在结果表中把没用的第一行直接扔掉，对爆破成功率零影响，只是让结果列表更清爽。 6.把资源池最大并发设为1，全程单线程排队发，禁止并行，防止同一秒内来多个请求，后来的 token 会被当场作废，都会变成403； 类比： 一次性门票： 闸机只认一次； 两个人同时把同一张票塞进闸机，机器不会放行任何一个人，而是直接报警作废。 7.Recursive grep（递归提取）： 只出现在Pitchfork / Cluster bomb类型 作用：从指定来源按正则取值，当成本次 payload。
