DICT (Dictionary Network Protocol，字典网络协议)

dict 协议是一个基于 TCP 的应用层协议。当你通过 SSRF 触发服务器访问 dict://<ip>:<port> 时，后端服务器（发起请求的一方）会尝试与目标 IP 和端口建立 TCP 连接。

  • 端口开放：如果目标端口开放且运行着服务，TCP 三次握手会成功。dict 协议通常会等待服务器发送“欢迎语”（Banner）。如果该端口运行的是其他协议（如 Redis、MySQL），它们在连接建立后往往会主动返回一些握手数据。
  • 端口关闭：如果端口关闭，目标主机会直接返回 RST 包，导致连接立即失败。
  • 过滤/丢弃：如果中间有防火墙，请求可能会超时。

结果： 攻击者通过观察后端服务器返回的错误信息、响应时间或具体的 Banner 内容（例如报错中包含 “Connection refused” 或返回了部分二进制字符），就能精准判断该端口的状态。

关于 dict://ip:port/info 这种形式，其实它是 dict 协议的一种快捷查询语法。

  • Banner 获取：在 SSRF 中，dict 协议最强大的地方在于它会回显。它不像某些协议（如 file://）只能读本地文件，dict 会把目标端口返回的初始信息（Banner 信息）直接抓取回来。
  • 举例： 如果你请求 dict://127.0.0.1:6379（Redis 默认端口），Redis 收到连接后会返回一个报错或状态。dict 协议会将这些字符串读取并展示给前端。这就实现了“信息泄露”。

注意： 关于你提到的 dict://ip:port/filename，这在标准的 dict 协议逻辑中其实是 dict://ip:port/lib:word 的变体。它并不是像 file:// 协议那样去磁盘上读取物理文件，而是试图向目标服务请求名为 filename 的词条。但在 SSRF 场景下，这更多被用来向内网服务（如 Redis）发送自定义指令。

核心结论： dict 协议之所以能查到这些信息，是因为它本质上是一个“话筒”。它在建立 TCP 连接后，会忠实地把对方发过来的任何声音（数据）传回给你，同时它也可以把你想说的话（指令）传给内网的其它服务。

DNS

攻击者利用了 DNS 协议里的一个参数：TTL (Time To Live，生存时间)。

  • 正常域名： TTL 通常是几十分钟或几小时。这意味着服务器解析一次后，会把结果“记住”（缓存），第二次访问时直接用缓存，不会再去问 DNS 服务器。
  • 攻击域名： 攻击者把 TTL 设置为 0。

关键过程就在这里：

  1. 第一次解析（检查）： 服务器问：“evil.com 是谁？” 攻击者的 DNS 服务器回答：“是 1.2.3.4（公网）”，并强调“我的有效期是 0 秒，别重用我”。服务器检查 1.2.3.4，发现合法。
  2. 第二次解析（访问）： 因为刚才的结果“有效期是 0 秒”，服务器不敢用旧数据，只能再次问一遍：“evil.com 是谁？”
  3. 攻击者的“变脸”： 就在这不到 1 秒的间隙里，攻击者的 DNS 服务器改口了：“现在它是 127.0.0.1（内网）了！”

FTP(File Transfer Protocol，文件传输协议)

该协议是Intemet文件传送的基础,它由一系列规格说明文档组成,目标是提高文件的共享性,提供非直接使用远程计算机,使存储介质对用
户透明和可靠高效地传送数据。

FTP服务器和客户端

FTP是一个客户/服务器系统。用户通过一个客户机程序连接至在远程计算机上运行的服务器程序。依照FTP
协议提供服务,进行文件传送的计算机就是FTP服务器,而连接FTP服务器,遵循FTP协议与服务器传送文件的电脑就是FTP客户端。

FTP用户授权

要连上FTP服务器(即“登陆”),必须要有该FTP服务器授权的帐号

FTP服务-工作方式

FTP支持两种模式,一种方式叫做Standard(也就是PORT方式,主动方式),一种是Passive(也就是PASV,被动方式)。

Standard模式
FTP的客户端发送PORT命令到FTP服务器。

Passive模式FTP的客户端发送PASV命令到FTP Server。

Port
FTP 客户端首先和FTP服务器的TCP 21端口建立连接,通过这个通道发送命令,客户端需要接收数据的时候在这个通道上发送PORT命令。
PORT命令包含了客户端用什么端口接收数据。在传送数据的时候,服务器端通过自己的TCP 20端口连接至客户端的指定端口发送数据。FTP
server必须和客户端建立一个新的连接用来传送数据。

Passive建立连接后发送的不是Port命令,而是Pasv命令。FTP服务器收到Pasv命令后,随机打开一
个高端端口(端口号大于1024)并且通知客户端在这个端口上传送数据的请求,客户端连接FTP服务器此端口,通过三次握手建立通道,然后
FTP服务器将通过这个端口进行数据的传送。
很多防火墙在设置的时候都是不允许接受外部发起的连接的,所以许多位于防火墙后或内网的FTP服务器不支持PASV模式,因为客户端无法
穿过防火墙打开FTP服务器的高端端口;而许多内网的客户端不能用PORT模式登陆FTP服务器,因为从服务器的TCP20无法和内部网络的客户端建立一个新的连接,造成无法工作。

HTTP（HyperText Transfer Protocol，超文本传输协议）

HTTP协议工作于客户端-服务端架构上，使用统一资源标识符(Uniform Resource Identifiers,URI)来传输数据和建立连接。URL(Uniform Resource Locator,统一资源定位符)是一种特殊类型的URI,包含了用于查找某个资源的足够的信息

浏览器作为HTTP客户端通过URL向HTTP服务端即WEB服务器发送所有请求。Web服务器根据接收到的请求后,向客户端发送响应信息。

完整的URL包含

例：http://www.aspxfans.com:8080/news/index.asp?boardID=5&ID=24618&page=1#name

1,协议部分:该URL的协议部分为“http:”,在Internet中可以使用多种协议

2.域名部分:该URL的域名部分为“www.aspxfans.com"。一个URL中,也可以使用IP地址作为域名使用

3.端口部分:跟在域名后面的是端口,域名和端口之间使用“:”作为分隔符。端口不是一个URL必须的部分,如果省略端口部分,将采用默认端口

4.虚拟目录部分:从域名后的第一个“/”开始到最后一个“/”为止,是虚拟目录部分。虚拟目录不是一个URL必须的部分。本例中的虚拟目录是“/news/”

5.文件名部分:从域名后的最后一个“/”开始到“?”为止,是文件名部分,如果没有“?”,则是从域名后的最后一个“/”开始到“#”为止,是文件部分,如果没有“?”和“#”,那么从域名后的最后一个“/”开始到结束,都是文件名部分。

本例中的文件名是“index.asp”。文
件名部分不是一个URL必须的部分,如果省略该部分,则使用默认的文件名

6.锚部分:从“#”开始到最后,都是锚部分。本例中的锚部分是“name”。

锚部分不是一个URL必须的部分

7.参数部分:从“?”开始到“#”为止之间的部分为参数部分,又称搜索部分、查询部分。本例中的参数部分为
“boardID=5&ID=24618&page=1”。参数可以允许有多个参数,参数与参数之间用“&”作为分隔符。

SSH(Secure Shell，安全外壳协议）

SSH是一种网络协议,用于计算机之间的加密登录。如果一个用户从这种登录是安全的,即使被中途截获,密码也不会泄露。

密码登录原理

1.远程Server收到Client端用户TopGun的登录请求,Server把自己的公钥发给用户

2.Client使用这个公钥,将密码进行加密。

3.Client将加密的密码发送给Server端。

4.远程Server用自己的私钥,解密登录密码,然后验证其合法性。

5.若验证结果,给Client相应的响应。

公钥免密登录原理

1.客户端向服务端发送连接请求，申请使用公钥认证方式登录

2.服务端收到请求后，会去该用户家目录的authorized_keys文件中检索，SSH 服务在解析authorized_keys文件时，如果发现某一行有乱码，那一行就会失效

如果没找到匹配的公钥，则退回到传统的密码登录模式

如果找到了对应的公钥，说明该客户端提前配置过互信，服务端发起“挑战”。

3.服务端在内存中随机生成一个字符串，使用刚才检索出来的客户端公钥，对随机字符串进行加密，生成一堆密文，然后把这堆密文发送给客户端

4.客户端收到这堆密文后，调用自己的私钥进行解密，成功还原出服务端的随机字符串，并将这个随机字符串结合当前会话的 Session ID，进行哈希运算生成一个数字签名（Signature），再把这个签名发送给服务端

5.服务端收到客户端发回来的签名后，用自己内存里原本就有的随机字符串和相同的算法也算一个签名，两边进行比对：

如果完全一致，则允许连接成功

ARP（Address Resolution Protocol，地址解析协议）

Flags(标志位)

这是判断目标是否“存活”的核心指标，它反映了 ARP 条目的状态：

  • 0x2 (ATF_COM)：完整 (Completed)。表示 ARP 请求已成功，系统已经拿到了对方的 MAC 地址。这意味着该主机目前是存活且可达的。
  • 0x0：未完成 (Incomplete)。表示系统发出了 ARP 请求，但对方没有回应（可能是 IP 不存在，也可能是对方禁用了 ARP 响应）。
  • 0x4 (ATF_PERM)：永久 (Permanent)。表示这是手动添加的静态 ARP 条目，不会自动过期。
  • 0x1 (ATF_INUSE)：表示该条目正在使用中（较少单独出现）。
